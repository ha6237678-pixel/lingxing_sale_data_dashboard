# 数据库交接与品线维度看板接入说明

生成日期：2026-06-28  
更新日期：2026-06-30

用途：给看板项目或另一台电脑上的 Codex 先阅读，用来理解当前 Supabase/PostgreSQL 数据库结构，并在原有“运营维度”看板基础上新增“品线维度”展示。

## 0. 这次看板要做什么

原看板只有运营维度，主要使用：

- `fact_operator_daily_metrics`
- `fact_settlement_profit`
- `dim_operator`
- `fact_operator_monthly_target`

现在数据库已经新增了品线维度，主要使用：

- `dim_product_line`
- `fact_product_line_daily_metrics`
- `fact_product_line_settlement_profit`

看板项目需要支持两套分析口径：

1. 运营维度：按运营负责人、运营小组看销售、广告、利润、目标完成。
2. 品线维度：按运营负责人 + 品线看销售、广告、流量、利润、费用率。

注意：品线数据不是从运营汇总数据里按比例拆出来的，而是领星接口按 `principal_uid + cid` 直接请求得到的真实品线口径数据。

## 1. 当前数据状态

| 表名 | 当前用途 | 数据范围/状态 |
|---|---|---|
| `dim_operator` | 运营负责人维度 | 17 个运营负责人 |
| `fact_operator_daily_metrics` | 运营维度产品表现 | `2026-01-01` 到 `2026-06-25` |
| `fact_settlement_profit` | 运营维度结算利润 | `2026-01-01` 到 `2026-06-17` |
| `fact_operator_monthly_target` | 运营月目标 | 当前已有 2026-04 到 2026-06 |
| `dim_product_line` | 品线维度主数据 | 以 `principal_uid + cid` 作为唯一业务键 |
| `fact_product_line_daily_metrics` | 品线维度产品表现 | `2026-01-01` 到 `2026-06-25`，共 176 天，13904 行，每天 79 行 |
| `fact_product_line_settlement_profit` | 品线维度结算利润 | `2026-01-01` 到 `2026-06-17`，共 168 天，12895 行，每天 75-78 行 |
| `crawl_run` | 爬虫运行日志 | 记录每次采集任务 |

品线产品表现每天 79 行，是因为源 Excel 中有 80 行配置，但存在重复的 `principal_uid + cid`，数据库业务唯一键会合并为 79 个有效品线组合。

品线结算利润每天 75-78 行是正常现象。领星接口对部分品线在某些日期会返回 `data=None`，表示该品线当天没有结算利润数据，不应当当成爬虫失败。

## 2. 核心关联关系

### 2.1 运营维度

```text
dim_operator.principal_uid
  -> fact_operator_daily_metrics.principal_uid
  -> fact_settlement_profit.principal_uid
  -> fact_operator_monthly_target.principal_uid
```

运营维度事实表唯一粒度：

```text
fact_operator_daily_metrics:
stat_date + principal_uid

fact_settlement_profit:
settlement_date + principal_uid + currency_code
```

### 2.2 品线维度

```text
dim_product_line.principal_uid + dim_product_line.cid
  -> fact_product_line_daily_metrics.principal_uid + fact_product_line_daily_metrics.cid
  -> fact_product_line_settlement_profit.principal_uid + fact_product_line_settlement_profit.cid
```

品线维度事实表唯一粒度：

```text
fact_product_line_daily_metrics:
stat_date + principal_uid + cid

fact_product_line_settlement_profit:
settlement_date + principal_uid + cid + currency_code
```

重要：看板关联品线时不能只用 `cid`。同一个 `cid` 可能出现在不同运营下面，必须使用 `principal_uid + cid`。

## 5. 指标计算口径

看板里不要直接平均行级比率，应该先聚合分子和分母，再计算比例。

### 5.1 产品表现指标

```sql
销售额 = sum(amount)
销量 = sum(volume)
订单商品数 = sum(order_items)
广告花费 = sum(spend)
广告销售额 = sum(ad_sales_amount)

广告费比 = sum(spend) / nullif(sum(amount), 0)
ACOS = sum(spend) / nullif(sum(ad_sales_amount), 0)
TACOS = sum(spend) / nullif(sum(amount), 0)
CVR = sum(order_items) / nullif(sum(sessions_total), 0)
广告CVR = sum(ad_order_quantity) / nullif(sum(clicks), 0)
自然CVR = sum(nature_order_items) / nullif(sum(nature_click), 0)
CTR = sum(clicks) / nullif(sum(impressions), 0)
CPC = sum(spend) / nullif(sum(clicks), 0)
```

说明：如果页面上同时展示“领星原始 ACOS/CVR”和“聚合重算 ACOS/CVR”，建议命名清楚。看板汇总场景优先使用聚合重算口径。

### 5.2 结算利润指标

```sql
结算销售额 = sum(total_sales_amount_with_tax)
广告成本 = sum(total_ads_cost)
平台费 = sum(platform_fee)
FBA配送费 = sum(fba_delivery_fee)
总成本 = sum(total_cost)
毛利 = sum(gross_profit)

广告费比 = abs(sum(total_ads_cost)) / nullif(sum(total_sales_amount_with_tax), 0)
平台费比 = abs(sum(platform_fee)) / nullif(sum(total_sales_amount_with_tax), 0)
FBA配送费比 = abs(sum(fba_delivery_fee)) / nullif(sum(total_sales_amount_with_tax), 0)
总成本占比 = abs(sum(total_cost)) / nullif(sum(total_sales_amount_with_tax), 0)
毛利率 = sum(gross_profit) / nullif(sum(total_sales_amount_with_tax), 0)
```

费用字段在结算利润里可能是负数或正数，前端展示“费用率”时建议用 `abs(sum(费用字段)) / 销售额`，展示金额明细时可以保留原始正负号。

## 6. 日期口径

产品表现和结算利润不是同一个刷新节奏：

| 数据 | 日期字段 | 当前完整范围 | 说明 |
|---|---|---|---|
| 运营产品表现 | `stat_date` | 到 `2026-06-25` | 日产品表现 |
| 运营结算利润 | `settlement_date` | 到 `2026-06-17` | 结算利润有延迟 |
| 品线产品表现 | `stat_date` | 到 `2026-06-25` | 日产品表现 |
| 品线结算利润 | `settlement_date` | 到 `2026-06-17` | 结算利润有延迟 |

看板如果把产品表现和结算利润放在同一张图里，需要注意日期范围不同。建议：

- 产品表现类卡片默认使用 `stat_date` 最新完整日期。
- 利润类卡片默认使用 `settlement_date` 最新完整日期。
- 如果做综合经营看板，日期范围建议取两类数据都完整的交集，例如当前到 `2026-06-17`。

## 7. 推荐 SQL

### 7.1 品线产品表现汇总

```sql
select
    stat_date,
    group_name,
    operator_name,
    principal_uid,
    product_line_name,
    cid,
    sum(amount) as sales_amount,
    sum(volume) as volume,
    sum(order_items) as order_items,
    sum(sessions_total) as sessions_total,
    sum(page_views_total) as page_views_total,
    sum(spend) as ad_spend,
    sum(ad_sales_amount) as ad_sales_amount,
    sum(spend) / nullif(sum(amount), 0) as ad_spend_rate,
    sum(spend) / nullif(sum(ad_sales_amount), 0) as acos,
    sum(order_items) / nullif(sum(sessions_total), 0) as cvr,
    sum(ad_order_quantity) / nullif(sum(clicks), 0) as ad_cvr,
    sum(nature_order_items) / nullif(sum(nature_click), 0) as nature_cvr
from fact_product_line_daily_metrics
where stat_date between date '2026-06-01' and date '2026-06-25'
group by stat_date, group_name, operator_name, principal_uid, product_line_name, cid
order by stat_date, group_name, operator_name, product_line_name;
```

### 7.2 品线结算利润汇总

```sql
select
    settlement_date,
    group_name,
    operator_name,
    principal_uid,
    product_line_name,
    cid,
    currency_code,
    sum(total_sales_amount_with_tax) as sales_amount,
    sum(total_ads_cost) as ad_cost,
    sum(platform_fee) as platform_fee,
    sum(fba_delivery_fee) as fba_delivery_fee,
    sum(total_cost) as total_cost,
    sum(gross_profit) as gross_profit,
    abs(sum(total_ads_cost)) / nullif(sum(total_sales_amount_with_tax), 0) as ad_cost_rate,
    abs(sum(platform_fee)) / nullif(sum(total_sales_amount_with_tax), 0) as platform_fee_rate,
    abs(sum(fba_delivery_fee)) / nullif(sum(total_sales_amount_with_tax), 0) as fba_delivery_fee_rate,
    sum(gross_profit) / nullif(sum(total_sales_amount_with_tax), 0) as gross_rate
from fact_product_line_settlement_profit
where settlement_date between date '2026-06-01' and date '2026-06-17'
group by settlement_date, group_name, operator_name, principal_uid, product_line_name, cid, currency_code
order by settlement_date, group_name, operator_name, product_line_name;
```

### 7.3 查每张表最新日期

```sql
select 'fact_operator_daily_metrics' as table_name, min(stat_date) as min_date, max(stat_date) as max_date, count(*) as rows
from fact_operator_daily_metrics
union all
select 'fact_settlement_profit', min(settlement_date), max(settlement_date), count(*)
from fact_settlement_profit
union all
select 'fact_product_line_daily_metrics', min(stat_date), max(stat_date), count(*)
from fact_product_line_daily_metrics
union all
select 'fact_product_line_settlement_profit', min(settlement_date), max(settlement_date), count(*)
from fact_product_line_settlement_profit;
```

### 7.4 检查品线重复业务键

```sql
select stat_date, principal_uid, cid, count(*)
from fact_product_line_daily_metrics
group by stat_date, principal_uid, cid
having count(*) > 1;

select settlement_date, principal_uid, cid, currency_code, count(*)
from fact_product_line_settlement_profit
group by settlement_date, principal_uid, cid, currency_code
having count(*) > 1;
```

正常结果应该为空。

## 8. 前端/后端接入建议

### 8.1 推荐做法

看板项目可以做一个统一维度切换：

```text
维度 = 运营 / 品线
```

当选择“运营”：

- 产品表现取 `fact_operator_daily_metrics`
- 结算利润取 `fact_settlement_profit`
- 维度字段使用 `group_name`、`operator_name`、`principal_uid`

当选择“品线”：

- 产品表现取 `fact_product_line_daily_metrics`
- 结算利润取 `fact_product_line_settlement_profit`
- 维度字段使用 `group_name`、`operator_name`、`principal_uid`、`product_line_name`、`cid`

### 8.2 不建议做法

不要把品线数据和运营数据简单 union 后当成同一张事实表直接聚合，因为两者粒度不同：

- 运营维度是一名运营一天一行。
- 品线维度是一名运营的一个品线一天一行。

如果把两者混在一起，容易重复计算销售额和利润。

### 8.3 API 返回字段建议

如果看板有后端 API，品线接口建议至少返回：

```text

operator_name
principal_uid
product_line_name
cid

cvr
ad_spend
ad_sales_amount
acos
tacos
nature_click
nature_order_items
nature_cvr
platform_fee
fba_delivery_fee
gross_profit
gross_rate
```

## 9. 数据安全和注意事项

1. 不要把 `.env`、Supabase 密码、`LINGXING_AUTH_TOKEN`、SMTP 密码写入 Git 或前端代码。
2. 前端不要暴露 Supabase service role key。
3. 如果前端直连 Supabase，需要确认 RLS 策略；如果通过后端 API 查询，建议由服务端持有数据库连接。
4. 中文名称只用于展示，不要用 `operator_name` 或 `product_line_name` 当唯一关联键。
5. 品线关联必须使用 `principal_uid + cid`。
6. 汇总比率必须聚合后重算，不要简单平均日级或行级比率。
7. 产品表现和结算利润日期范围不同，综合展示时要明确用哪个日期口径。
8. 结算利润部分品线缺行通常是业务无数据，不一定是爬虫失败。

## 10. 给看板项目 Codex 的任务顺序

1. 先阅读本文档，理解运营维度和品线维度是两套不同粒度。
2. 检查当前看板获取数据的位置，是前端直连 Supabase，还是后端 API 查询。
3. 保留原运营维度页面逻辑。
4. 新增品线维度查询逻辑，读取 `fact_product_line_daily_metrics` 和 `fact_product_line_settlement_profit`。
5. 新增筛选项：`product_line_name`、`cid`。
6. 所有费比、ACOS、CVR、毛利率统一使用聚合后重算。
7. 处理产品表现和结算利润最新日期不一致的问题。
8. 做一轮 SQL 或接口校验，确认汇总数不会因为 join 或 union 重复计算。
