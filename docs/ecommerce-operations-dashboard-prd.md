# 电商运营数据看板 PRD

## 1. 项目概述

### 1.1 项目背景

公司主营亚马逊汽配改装类目，SKU 数量多，每个运营通常按品线负责业务。当前阶段看板暂不下钻到品线和 SKU 维度，优先支持按时间、组别、运营负责人查看经营数据。

目前数据库已建设完成，数据已落入 Supabase/PostgreSQL。GitHub Actions 已部署每日定时爬虫任务，可自动采集并更新销售、流量、广告、结算利润和月度目标相关数据。

因此，本项目当前重点从“数据准备”转为“数据消费与可视化展示”：看板只需要连接现有数据库，基于已生成的数据表进行查询、聚合和展示。

### 1.2 项目目标

建设一个 PC 端电商运营数据看板，用于运营部门领导和运营人员日常监控销售、流量、广告、利润和目标完成情况，帮助快速发现异常、比较组别与个人表现、跟踪月度目标达成。

### 1.3 当前建设状态

| 模块 | 状态 | 说明 |
| --- | --- | --- |
| 数据库 | 已完成 | Supabase/PostgreSQL 已生成核心业务表 |
| 数据采集 | 已完成 | GitHub Actions 已部署每日定时爬虫 |
| 数据更新 | 已完成 | 爬虫每日自动更新数据库 |
| 失败通知 | 已完成 | GitHub Actions 通过 SMTP Secret 发送失败邮件 |
| 看板前端 | 待开发 | 需要调用数据库展示指标 |
| 数据接口/查询层 | 待开发 | 可直接连接数据库或封装 API |
| 权限控制 | 一期简化 | 当前需求为所有人可见 |

### 1.4 项目范围

一期范围：

- 连接现有 PostgreSQL 数据库
- 销售数据总览
- 流量与广告数据总览
- 利润数据总览
- 目标完成率监控
- 时间、组别、运营筛选
- 数据更新时间、采集状态展示

暂不包含：

- SKU 维度分析
- 品线维度分析
- 预测模型
- 复杂权限隔离
- 移动端适配
- 爬虫系统重构

## 2. 目标用户

| 用户角色 | 使用目的 | 典型场景 |
| --- | --- | --- |
| 运营部门领导 | 查看整体经营表现、组别对比、目标完成和异常情况 | 每日晨会、周会、月度复盘 |

## 3. 数据架构与自动化更新说明

### 3.1 当前数据链路状态

当前项目的数据采集与数据库写入链路已完成配置。

数据通过 GitHub Actions 每日定时运行爬虫任务，爬虫从领星/亚马逊相关数据源采集销售、流量、广告、结算利润等数据，并自动写入 Supabase/PostgreSQL 数据库。

看板项目当前不需要再实现数据采集逻辑，只需要连接现有数据库并进行查询、聚合和可视化展示。

### 3.2 GitHub Actions Secrets 配置

当前 GitHub Actions 已配置 Repository Secrets，用于支持每日定时爬虫、数据库写入和失败邮件通知。

已配置的 Secrets 包括：

| Secret 名称 | 用途 |
| --- | --- |
| `DATABASE_URL` | PostgreSQL/Supabase 数据库连接地址，用于爬虫写入数据及后端读取数据 |
| `LINGXING_AUTH_TOKEN` | 领星接口或爬虫认证 Token |
| `SMTP_HOST` | 邮件服务 SMTP 地址 |
| `SMTP_PORT` | 邮件服务 SMTP 端口 |
| `SMTP_USERNAME` | 邮件发送账号 |
| `SMTP_PASSWORD` | 邮件发送账号密码或授权码 |
| `MAIL_TO` | 爬虫失败或异常时的邮件接收人 |

注意：文档和代码仓库只记录 Secret 名称与用途，不记录任何真实密钥、密码、Token 或数据库连接串。

### 3.3 自动化更新流程

```text
GitHub Actions 定时触发
    -> 读取 Repository Secrets
    -> 使用 LINGXING_AUTH_TOKEN 执行数据采集
    -> 清洗并转换采集结果
    -> 使用 DATABASE_URL 写入 Supabase/PostgreSQL
    -> 写入 crawl_run 采集日志
    -> 如任务失败，使用 SMTP 配置发送邮件到 MAIL_TO
    -> 看板读取数据库最新数据并展示
```

### 3.4 看板侧职责边界

看板侧负责：

- 通过后端服务读取数据库数据
- 对销售、流量、广告、利润和目标数据进行聚合查询
- 展示指标卡、趋势图、排行、明细表和数据状态
- 展示最近数据更新时间和爬虫执行状态
- 在数据库连接失败、数据为空或数据延迟时给出提示

看板侧不负责：

- 不负责执行爬虫
- 不负责维护领星 Token
- 不负责发送失败通知邮件
- 不负责直接管理 GitHub Actions Secrets
- 不在前端暴露任何 Secret 或数据库连接信息

### 3.5 核心数据表

| 表名 | 用途 | 粒度 |
| --- | --- | --- |
| `dim_operator` | 运营负责人维表 | 一个运营负责人一行 |
| `fact_operator_daily_metrics` | 销售、流量、广告日指标 | 日期 + 运营负责人 |
| `fact_settlement_profit` | 结算利润日指标 | 结算日期 + 运营负责人 + 币种 |
| `fact_operator_monthly_target` | 月度销售/利润目标 | 月份 + 运营负责人 |
| `crawl_run` | 爬虫采集日志 | 一次采集任务一行 |

### 3.6 核心关联键

```text
principal_uid
```

所有事实表应通过 `principal_uid` 与运营维表关联，不应仅使用运营姓名关联。

## 5. 功能用户故事与验收标准

### 5.1 数据库连接与数据读取

**用户故事**

作为看板使用者，我希望系统直接读取已经生成好的数据库数据，以便无需手工导入 Excel 或重复处理数据。

**验收标准**

- 系统通过服务端环境变量 `DATABASE_URL` 连接 PostgreSQL。
- 前端不得直接暴露数据库连接串。
- 数据查询成功后，页面能正常展示销售、流量、广告、利润和目标数据。
- 数据库连接失败时，页面展示友好错误提示。
- 查询逻辑支持日期、组别、运营筛选。
- 所有敏感配置均通过环境变量或部署平台 Secret 管理。

### 5.2 全局筛选器

**用户故事**

作为运营领导，我希望按日期、组别、运营负责人筛选数据，以便快速查看不同范围内的经营表现。

**验收标准**

- 支持日期范围筛选。
- 支持组别筛选。
- 支持运营负责人筛选。
- 组别和运营负责人筛选支持联动。
- 筛选条件变化后，所有指标卡、趋势图、排行和表格同步刷新。
- 默认展示最近可用数据周期。
- 筛选为空时有明确提示，不展示错误数据。

### 5.3 销售数据总览

**用户故事**

作为运营人员，我希望看到销售额、销量、订单量、广告订单占比和 B2B 销售表现，以便判断销售基本盘是否健康。

**验收标准**

- 数据来源为 `fact_operator_daily_metrics`。
- 展示销量、销售额、订单量、广告订单量、广告订单占比、B2B 销量、B2B 销售额、B2B 订单量。
- 广告订单占比公式为 `sum(ad_order_quantity) / sum(order_items)`。
- 比率分母为 0 时展示为 `0` 或 `-`。
- 指标卡、趋势图、排行均受筛选器控制。
- 金额、数量、百分比格式统一。

### 5.4 流量与广告数据总览

**用户故事**

作为运营人员，我希望同时查看流量、广告投入和转化效率，以便判断销售变化是由流量、广告还是转化导致。

**验收标准**

- 数据来源为 `fact_operator_daily_metrics`。
- 展示 Sessions、PV、CVR、广告展示、广告点击、CTR、CPC、广告花费、广告 CVR、广告销售额、ACOS、TACOS、自然点击量、自然订单量、自然 CVR。
- 比率类指标按聚合后的分子/分母计算。
- 支持广告花费与广告销售额对比图。
- 支持 ACOS/TACOS 趋势图。
- 无数据日期应展示为空或断点，不得错误补零误导用户。

### 5.5 利润数据总览

**用户故事**

作为运营领导，我希望查看结算利润、费用占比和成本结构，以便判断销售增长是否带来了真实利润。

**验收标准**

- 数据来源为 `fact_settlement_profit`。
- 展示结算销售额、销量、广告销售额、广告销量、毛利润、毛利率、退款率、平台费、FBA 发货费、广告费、推广费、仓储费、采购成本、头程成本、合计成本。
- 毛利率公式为 `sum(gross_profit) / sum(total_sales_amount_with_tax)`。
- 支持费用结构展示。
- 支持按组别、运营查看利润排行。
- 币种字段 `currency_code` 可展示，默认以当前主要币种 USD 汇总。

### 5.6 目标完成率

**用户故事**

作为运营领导，我希望看到销售目标和利润目标完成情况，以便判断本月目标是否有达成风险。

**验收标准**

- 数据来源为 `fact_operator_monthly_target`、`fact_operator_daily_metrics`、`fact_settlement_profit`。
- 展示销售目标、实际销售额、销售完成率。
- 展示利润目标、实际毛利润、利润完成率。
- 支持月化完成率。
- 目标缺失时展示“未配置目标”，不得按 0 计算误导用户。
- 人员维表缺失 `principal_uid` 时应提示数据不可关联。

### 5.7 数据刷新状态

**用户故事**

作为数据维护人员，我希望看到数据最近更新时间和采集任务状态，以便判断看板数据是否可信。

**验收标准**

- 展示销售/流量/广告数据最大 `stat_date`。
- 展示利润数据最大 `settlement_date`。
- 展示目标数据最大 `target_month`。
- 展示最近一次 `crawl_run` 状态。
- 采集失败或部分失败时展示错误摘要。
- 数据延迟超过预设阈值时给出提示。
- 页面不得在数据未更新时仍显示“今日最新”之类误导文案。

### 5.8 明细表格

**用户故事**

作为运营人员，我希望查看当前筛选范围下的明细数据，以便核对图表背后的具体数值。

**验收标准**

- 支持按运营、组别、日期展示明细。
- 支持排序。
- 支持分页。
- 支持导出当前筛选结果。
- 指标口径与上方汇总指标一致。
- 表格加载中、无数据、加载失败均有明确状态。

### 5.9 指标说明

**用户故事**

作为看板使用者，我希望查看指标定义和公式，以便理解每个指标的含义。

**验收标准**

- 指标卡和表头支持悬停查看说明。
- 比率类指标展示公式。
- 明确区分销售侧数据和结算利润侧数据。
- 明确提示销售额字段存在不同口径：销售表现表使用 `amount`，利润表使用 `total_sales_amount_with_tax`。
- 指标说明与数据库字段保持一致。

## 6. 关键指标公式

| 指标 | 公式 |
| --- | --- |
| 广告订单占比 | `sum(ad_order_quantity) / sum(order_items)` |
| CVR | `sum(order_items) / sum(sessions_total)` |
| CTR | `sum(clicks) / sum(impressions)` |
| CPC | `sum(spend) / sum(clicks)` |
| ACOS | `sum(spend) / sum(ad_sales_amount)` |
| TACOS | `sum(spend) / sum(amount)` |
| 广告 CVR | `sum(ad_order_quantity) / sum(clicks)` |
| 自然 CVR | `sum(nature_order_items) / sum(nature_click)` |
| 毛利率 | `sum(gross_profit) / sum(total_sales_amount_with_tax)` |
| 广告费占比 | `sum(total_ads_cost) / sum(total_sales_amount_with_tax)` |
| 平台费占比 | `sum(platform_fee) / sum(total_sales_amount_with_tax)` |
| FBA 发货费占比 | `sum(fba_delivery_fee) / sum(total_sales_amount_with_tax)` |
| 仓储费占比 | `sum(total_storage_fee) / sum(total_sales_amount_with_tax)` |
| 采购成本占比 | `sum(total_cost) / sum(total_sales_amount_with_tax)` |
| 销售目标完成率 | `本月累计销售额 / 月销售目标` |
| 利润目标完成率 | `本月累计毛利润 / 月利润目标` |
| 月化销售完成率 | `本月累计销售额 / 已过天数 * 当月总天数 / 月销售目标` |
| 月化利润完成率 | `本月累计毛利润 / 已过天数 * 当月总天数 / 月利润目标` |

## 8. 非功能要求

### 8.1 性能要求

- 首屏加载时间建议控制在 3 秒以内。
- 常规筛选响应时间建议控制在 1 秒以内。
- 聚合计算应尽量在数据库或服务端完成。
- 明细表格必须分页加载。
- 常用筛选字段应建立索引，例如日期、`principal_uid`、`group_name`。
- 趋势图、排行和指标卡应避免重复请求同一数据。
- 大范围查询应使用后端聚合，避免前端加载全量明细后计算。

### 8.2 安全要求

- GitHub Actions 已使用 Repository Secrets 管理爬虫 Token、数据库连接和邮件通知配置。
- 看板服务端通过环境变量读取数据库连接信息。
- 前端不得读取或暴露 `DATABASE_URL`。
- 前端不得读取或暴露 `LINGXING_AUTH_TOKEN`。
- SMTP 配置仅用于 GitHub Actions 爬虫失败通知，不应进入看板前端。
- 日志中不得打印数据库密码、Token、SMTP 密码等敏感信息。
- 数据库查询接口需要校验参数，防止 SQL 注入。
- 如后续部署看板后端，应在部署平台单独配置 `DATABASE_URL` 环境变量。
- 当前已暴露过的密钥建议立即轮换。

### 8.3 兼容性要求

- 一期优先支持 PC 端。
- 推荐兼容最新版 Chrome、Edge。
- 页面应适配常见办公显示器分辨率。
- 图表和表格在 1366px 宽度下不得遮挡或错位。
- 后续如需移动端，应另行补充响应式设计要求。

### 8.4 可维护性要求

- 指标字段需统一中英文命名。
- 指标公式应集中维护。
- 数据库字段、接口字段、前端展示字段应建立映射。
- 新增指标时应同步更新接口文档、测试用例和指标说明。
- 关键聚合逻辑应有自动化测试。

## 9. 数据质量要求

- `fact_operator_daily_metrics` 唯一键：`stat_date + principal_uid`。
- `fact_settlement_profit` 唯一键：`settlement_date + principal_uid + currency_code`。
- `fact_operator_monthly_target` 唯一键：`target_month + principal_uid`。
- 所有事实表应优先通过 `principal_uid` 关联。
- 核心维度 `principal_uid` 必须完整。
- 运营姓名变更时，不影响历史事实表关联。
- 目标数据缺失时必须提示。
- 数据采集失败时必须记录到 `crawl_run`。
- 页面必须展示数据最新日期，避免用户误解为实时数据。
- 比率分母为 0 时统一处理为 `0` 或 `-`。

## 11. 验收标准总览

- GitHub Actions 已能通过 Repository Secrets 完成每日定时采集、数据库写入和失败邮件通知。

- 能展示销售、流量、广告、利润和目标完成率。

## 12. 风险与待确认事项

| 风险/问题 | 说明 | 建议 |
| --- | --- | --- |
| 品线维度暂不做 | 原始业务提到运营负责品线，但一期暂不需要 | 后续扩展品线维表和品线事实表 |
